General Information:

Most entities capable of inflicting damage have a "Kill Description" keyvalue. This determines what the entity is called if it kills a scientist. The usual string for this is like "Walter was killed by xxxx" where xxxx would be the kill description. Each entity has a default if it is left blank. The string may look different depending on what kind of damage the entity does. For example func_doors will say crushed and info_shootpoints will say shot.
Special case is the trigger_hurt. Since it isn't really an object of substance, it cannot simply define what it is. From the players perspective it doesn't even exist, it is simply an effect of some event that causes damage. You can type any message as to what happened to the scientist. By default the scientist name is placed in front of the killdesc string, with a space in between (ie. Walter xxxx), however if you put a "%s" into the string, then that will be replaced with the scientist name instead (for example if the string is "The devil ate %s", the death message would be "The devil ate Walter").

For any entity with a "Master" value, if you put a ! infront of the master targetname, then the master relationship will be inverted. The entity will work when the master is off and gets disabled when the master is turned on.

Using the command fire "targetname", where targetname is the name of the entity to trigger (no quotes), allows you to trigger entities from the console. Very useful for testing effects. Examle: to fire a func_train named crusher type:
fire crusher

beam_cannon: Beam Effect
  • targetname: Name
  • Render FX
  • killdesc: Kill Description
  • LaserTarget: Target of Laser
      Leave this blank if the beam is to be used by a beam_turret or an info_shootpoint_beam
  • rendermode: Fade Ends
    • None
    • Start
    • End
    • Both
  • renderamt: Brightness (1 - 255)
  • rendercolor: Beam Color (RGB)
  • width: Width of beam (pixels*0.1)
      A very wide beams will get larger scorch decals if enabled
  • flicker: Scale flicker (0-1)
      Randomly change the width by this much. This is a multiplier. 0 is a stable beam, 1 randomizes from 0-200%.
  • NoiseAmplitude: Amount of noise (0-255)
      In spiral mode, this is the width of the spiral effect
  • texture: Beam Sprite
  • StartSprite: Start Sprite
  • StartSpriteScale: Start Scale
  • slight: Start DLight
      0 0 0 or blank is no light
  • EndSprite: End Sprite
  • EndSpriteScale: End Scale
  • elight: End DLight
      0 0 0 or blank is no light
  • TextureScroll: Texture Scroll Rate (0-100)
  • framestart: Starting Frame
  • damage: Damage / second
  • damagetype: Damage Type
  • radius: Damage Radius
      If this is above 0, the beam will do area-effect damage.
  • EndSprite: End Sprite
    • None
    • Explosion
        One explosion when the beam is turned on.
    • Explosions
        Continual explosions at End FX Value intervals,
    • Sprites
        Leaves a trail of EndSprites instead of one that moves with the beam.
    • Beam Torus/Disk/Cylinder
        These are not expected to work
    • Particles
        Should create a particle spray. Not tested.
    • Blood
        Splatters human blood. For whatever effect that would be useful.
    • Alien Blood
        Splatters alien blood. For whatever effect that would be useful.
  • spawnflags:
    • Start On
    • Point2Point
        Beam is rendered all the way to target entity even if it hits something before it.
    • Directional
        Beam will continue until it hits something, even past the target entity if its nonsolid.
    • Spiral
        Spiral beam like the gluon gun. Must use NoiseAmplitude to set the width of the spiral.
    • StartSparks
    • EndSparks
    • Decal End
    • Glow Sprites
    • Add Sprites
        Normally, the start sprite is glow type and end sprite is additive, these allow setting both to same rendermode.
    • Solid
        Beam is rendered opague. Start & end sprites will be rendered in texture rendermode, unless the above spawnflags are used.
    • Ignore Glass
        Beam traceline will go through non-static texture-mode brushes. (Breakables, doors, etc. but not func_walls or unbreakable func_breakables)
    • Mimic target
        Duplicates the movement and on/off status of another beam_cannon. Specify the other beam in LaserTarget. Both ends will be matched.

beam_multi: Multi Beam Effect
This entity creates an array of identical beams (up to 16). Each beam has the properties of a beam_cannon.
  • targetname: Name
  • A HREF="#renderfields">Render FX
  • killdesc: Kill description
  • start: Start entity(s)
      One or more entities with this targetname will be used as start points for the beams.
  • end: End entity(s)
      One or more entities with this targetname will be used as end points for the beams in some multi types.
  • type: Multi type
    • Point->Multiple
        One start entity is used with multiple end entities.
    • Multiple->Point
        As above but inverse.
    • Point Grid
        Multiple start points and multiple end points. Also known as web mode. Creates a random pattern.
    • Target Pairs
        Uses multiple start entities, and each beam goes from it's start entity to the entity defined as the start entity's "target".
    • Target String
        Uses one start entity, beam goes from the start entity to the entity defined as the start entity's target, and the next beam starts from that
    • Random Point
        Uses multiple start points, and places the end points of the beams out in random directions from those points.
    • Random Area
        Uses multiple start points, and places both the start and end points of the beams out in random directions from those points.
  • beamcount: Number of beams (max 16)
      Number of beams 2-16. For point->multiple mode, this can't be more than the number of end entities. For multiple->point and target pairs, this can't be more than the number of start entities. No restrictions in other modes.

beam_turret: Beam Turret System
This entity isn't actually a turret. It's an intangible entity that can control and aim a beam_cannon. The beam is aimed as if it was at the location of the beam_turret, but fired from the beam_cannon location, so if they are not the same, the beam is likely to miss.
  • targetname: Name
  • angles: Pitch Yaw Roll (Y Z X)
  • Allies: Allies
      Defines who this turret does NOT shoot.
    • None
    • Player
    • GoodGuys
    • Human Military
    • Aliens
    • All but player
    • All but scientists
  • type: Movetype
    • Normal
        Beam fires at one spot until duration runs out.
    • Slash
        Beam is fired a slighly away from the target and makes a slashing move across the target.
    • Shots
        Beam is fired multiple times at the same target.
    • Sweep
        Beam tracks its target. If target is killed it will switch to another one.
  • target: Beam Entity
      Must be a valid beam_cannon.
  • netname: Charge Entity
      If you want to use a charge effect, this must be a valid beam_charge.
  • shots: Number of shots
      How many times the beam is refired in "Shots" mode.
  • Duration: Beam Duration
      Duration of firing sequence once the beam is online.
  • wait: Recharge time
      How long it takes until it fires again. Randomized to 100-130%.

      How large the slash move is in "Slash" mode

func_turret: Advanced Brush Gun Turret
A func_turret is most typically used as a gun turret. It is like a func_tank with no firing code. You can't make it shoot by itself. To make it fire, you use an info_shootpoint. These intangible, linked point entities define where the turret shoots from. One turret can control any number of shootpoints of any type.
NOTE: You don't have to use shootpoints with these. You could use them as security cameras, or you could do something else with it. In all its simplicity, a func_turret is a brush entity that tracks targets and triggers entities when its "on target".

      Primary & secondary are triggered on when the turret is pointing at its target (usually to start firing), and off when it is not (to stop firing). You can trigger any entities from these. On a player controlled turret, these match the players primary & secondary fire button.

This crane entity picks up and transports entities along a path_corner path. You need at least one "wait for retrigger" and one "teleport" path_corner in the path. The crane moves along the path until it encounters a "wait for retrigger". It will then wait at that position, checking the ground underneath the position of the next "teleport" node. When an entity is on the ground under the "teleport" path_corner, the crane will start to move again. It will try to pick up the entity at the "teleport" node. entity_link: Entity Multilinker
Universal Entity Multilinker. Yep, this one will bind any 2 entities together. The only minor exception is beams, which can only be position-linked, not angle-linked but thats what the info_shootpoint_beam is for. Of course, there are a lot of entities which just aren't worth linking.
It is important to note that the positional link will happen at the location of this entity. ie. all child entities will be moved to the location of the entity_link and then they follow the parent entity at the same relative location as the entity_link had in the map file.
  • target: Target
      This defines the child entities for the link. You can have more than one, but they will all be linked to the same point.
  • angles: Pitch Yaw Roll (Y Z X)
  • message: Link Parent
      This defines the parent entity for the link.
  • spawnflags:
    • Link X Rot
    • Link Y Rot
    • Link Z Rot
        These allow you to link rotation around the 3 world axes.
    • Full Pos Link
        This one determines complete positional linkage. It considers the parent entity's rotation and movement, and links the child entities XYZ location.
    • Link X Pos
    • Link Y Pos
    • Link Z Pos
        These allow linking the XYZ axis movements independently. Note that an entity cannot move on its own on an axis that is linked, so that is why these are provided. They do NOT consider parent rotation, only movement.

Lighting Flags:


 Zoner's Halflife Tools 2.2.0 and newer has support for flags that affect the lighting of brush entities. The most important thing about this is that by setting zhlt_lightflags to 2 will make a brush entity block light. I have included the flag keyvalue in the FGD in some entities that I have used it in myself. There are 2 other flags aswell, read ZHLTReference.html that comes with ZHLT to learn more.

Render Properties:


 Almost all visible entities can have their appearance changed by setting 4 keyvalues (rendermode, renderfx, rendercolor, renderamt). Rendermodes:
  • Normal
      Normal mode is completely opague. All the other values are ignored, except for the last 3 renderfx types)
  • Color
      BSP solids are rendered untextured, with the color defined in rendercolor. Only affects BSP solids, others will render as in texture mode.
  • Texture
      Standard textured transparent mode. Use renderamt to define transparency.
  • Glow
      Sprite is rendered as a lens flare glow. Essentially like additive, but it becomes larger and less bright as the player gets farther. The sprite will be visible through geometry, as long as the player has a clear line of sight to the origin of the sprite. On non-sprite entities this is just additive.
  • Solid
      For BSP solids, makes textures beginning with { show their transparency correctly (blue parts are invisible).
  • Additive
      The color of the object is added to the color of the background. This effectively means that black areas are invisible and white areas are fully opague. Creates the impression that the object is pure energy or light.
Renderfx:
  • Normal
      No FX
  • Slow Pulse
  • Fast Pulse
  • Slow Wide Pulse
  • Fast Wide Pulse
  • Slow Strobe
  • Fast Strobe
  • Faster Strobe
  • Slow Flicker
  • Fast Flicker
  • Slow Fade Away
  • Fast Fade Away
  • Slow Become Solid
  • Fast Become Solid
      All these pulse/strobe/fade effects require non-normal rendermode.
  • Constant Glow
      Only affects glow rendermode on sprites. Disables distance fade and size increase.
  • Distort
      Distortion seems to work best on models. Sprites and BSP solids just flicker.
  • Hologram (Distort + fade)
      Hologram is a distort combined with distance fade. Only seems to work right with models. Use simple distort if you need a hologram sprite/brush entity.
  • Strecth Sideways
      This is the effect used for the garg death. I've never used it anywhere. Makes things look fat if viewed from the front or back, but from the side they look kinda flat. Only works on .mdl models
  • Glowing Shell
      This is my favorite renderfx. It creates that "shield" effect around a model. To use it, set rendercolor to the color of the shell and renderamt to the distance of the shell from the model. Only works on .mdl models.
Renderamt is the transparency (or actually opacity) of the entity. It is used in all rendermodes except Normal. 0 is invisible, 255 is fully visible.

Rendercolor is the color used for the glowshell fx, the color rendermode in brush entities, beam entity color and sprite color. Sprites can be "recolored" with it in any rendermode. The sprite color will be multiplied with the rendercolor so a blue sprite and a red rendercolor will be black not red. Beam entities MUST have color (not 0 0 0) or they will be invisible. If you want a beam with the default color from the .spr file, use 255 255 255 as the color. Sprites will get default color with 0 0 0.
